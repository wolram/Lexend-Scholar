# OKRs Q1 2026 — Lexend Scholar

**Período**: Janeiro — Março 2026
**Review**: Semanal (sextas-feiras, 30 min)
**Retrospectiva**: Final de março 2026
**Owner**: Marlow Sousa (Founder)

## Contexto

O Q1 2026 é o trimestre de lançamento e validação do MVP do Lexend Scholar. O foco é:
1. Lançar um produto funcional e estável
2. Onboardar escolas piloto reais e aprender com elas
3. Validar que o modelo de negócio é viável

---

## Objetivo 1: Lançar MVP funcional e onboardar escolas piloto

**Por que**: Precisamos de escolas reais usando o produto para validar hipóteses e gerar feedback que guie o desenvolvimento. Sem uso real, estamos construindo em um vácuo.

**KR 1.1**: 5 escolas piloto onboardadas e ativas (ao menos 1 turma criada, frequência lançada em pelo menos 3 semanas consecutivas)
- Baseline: 0 escolas
- Meta: 5 escolas
- Progresso: _ / 5

**KR 1.2**: NPS de experiência de produto ≥ 35 (medido no D+30 de cada escola)
- Baseline: não medido
- Meta: NPS ≥ 35
- Progresso: _

**KR 1.3**: Feature core completa: Frequência + Notas + Secretaria básica (emissão de documentos) funcionando em iOS e Web
- Baseline: frequência em iOS (parcial)
- Meta: 3 módulos completos em ambas as plataformas
- Progresso: _ / 3 módulos

**KR 1.4**: Zero incidentes P0 (sistema completamente fora do ar) durante o período de piloto
- Baseline: não medido
- Meta: 0 incidentes P0
- Progresso: monitorar via Sentry/Better Uptime

---

## Objetivo 2: Validar modelo de negócio

**Por que**: Validar que escolas estão dispostas a pagar pelo produto e que o modelo de pricing faz sentido antes de investir em crescimento acelerado.

**KR 2.1**: 3 contratos pagos assinados (mínimo 3 meses, qualquer plano)
- Baseline: 0 contratos
- Meta: 3 contratos
- Progresso: _ / 3

**KR 2.2**: MRR (Monthly Recurring Revenue) ≥ R$3.000 ao final de março
- Baseline: R$0
- Meta: R$3.000
- Progresso: R$_

**KR 2.3**: Taxa de conversão de trial para pago ≥ 40% (escolas que testaram e decidiram pagar)
- Baseline: não medido
- Meta: ≥ 40%
- Progresso: _ / _ (X de Y escolas converteram)

**KR 2.4**: Churn de clientes pagos = 0 no Q1 (nenhum cliente que pagou cancelou)
- Baseline: N/A
- Meta: 0 cancelamentos
- Progresso: monitorar

---

## Objetivo 3: Construir base técnica sólida

**Por que**: Crescer sobre uma base técnica frágil é um multiplicador de problemas. Precisamos de fundações sólidas antes de escalar.

**KR 3.1**: Cobertura de testes automatizados ≥ 60% (crítico: fluxos de matrícula, frequência, notas e financeiro)
- Baseline: ~15%
- Meta: 60%
- Progresso: _%

**KR 3.2**: Uptime da API ≥ 99,5% durante todo o Q1 (medido via Better Uptime)
- Baseline: não medido
- Meta: ≥ 99,5%
- Progresso: _% (link: betteruptime.com/status/...)

**KR 3.3**: Tempo de resposta médio da API ≤ 300ms no p95 (medido via Vercel Analytics)
- Baseline: não medido
- Meta: ≤ 300ms p95
- Progresso: _ms

**KR 3.4**: Pipeline de CI/CD completo: testes automáticos passando antes de qualquer merge para main
- Baseline: CI básico
- Meta: CI/CD completo com testes, lint, type check e deploy automático para staging
- Progresso: Em andamento / Concluído
